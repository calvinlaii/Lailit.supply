## Specific Question Decisions

*(These answer the five specific questions from the phase scope brief.)*

### Q1 — DashboardNav: Top bar or sidebar?

**Decision: Top bar.** Same structural pattern as `TopNav` from Phase 1.

Rationale: The minimal stub warrants zero sidebar infrastructure. Phase 5 builds the full dashboard grid — at that point, the pattern evolves if needed. Committing to a sidebar now would create dead UI. The top bar pattern is already understood by the codebase and users. Phase 5 builders inherit this decision explicitly.

**DashboardNav layout contract:**
- Height: 64px (`h-16`)
- Background: white (`bg-white`)
- Bottom border: 1px `border-neutral-200`
- Position: `sticky top-0 z-50` (same as `TopNav`)
- Left slot: logo wordmark `lailit.supply` — 16px, weight 600, links to `/dashboard`
- Right slot: user email (label size, muted) + avatar circle + dropdown trigger chevron (`ChevronDown` from lucide, 16px)
- Padding: `max-w-[1200px] mx-auto px-4 sm:px-8 lg:px-12` (matches `TopNav` container)

**Phase 5 forward-compatibility note:** When Phase 5 builds the full dashboard, it may add a sidebar. If so, `DashboardNav` top bar remains as the top rail; the sidebar is additive below it. Nothing in this spec prevents a sidebar. The top bar does not include a "hamburger to open sidebar" slot in Phase 2 — Phase 5 can add that.

### Q2 — Loading state on LoginForm submit: Inline spinner in button or full-page loader?

**Decision: Inline spinner inside the "Kirim Magic Link" button.** The button text swaps to a spinner + "Mengirim..." while the Server Action is in-flight.

Rationale: Full-page loaders for a form submit are jarring and feel mobile-app rather than web. An inline spinner scopes the feedback precisely to the action in progress — consistent with modern form UX and appropriate for the editorial restraint of this design system. The button width does not change (no layout shift).

**Loading state contract:**
- Button background: `neutral-800` (slightly lighter than resting `neutral-950` — visually communicates "busy")
- Button content: `<Loader2 size={16} className="animate-spin mr-2" />` + `Mengirim...`
- Button: `disabled` attribute set, `cursor-not-allowed`
- Email input: `disabled` attribute set (prevents edit mid-flight)
- Duration: until Server Action resolves (success or error)
- No timeout UI — the Server Action is fast; do not add artificial delay

**Transition into success state:**
- Button and input re-enable
- Form is replaced by the success message block (see Copywriting Contract below)
- No page navigation — the success message lives in the same card

### Q3 — Avatar initial: Size, shape, color treatment

**Decision:** 32×32px circle, filled `neutral-950`, white initial letter, no border.

Rationale: In a pure black/white system, a filled circle with the initial as white text on `neutral-950` is the only choice that reads as an avatar without introducing any new color. Square avatars belong to product logos; circle is the universal "person" signal. 32px at label size (16px initial) is large enough to read in a 64px nav rail without crowding the email text beside it.

**Avatar contract:**
```
Width: 32px (w-8)
Height: 32px (h-8)
Shape: circle (rounded-full)
Background: neutral-950
Text: first character of user email, uppercase, white, 16px, weight 600
Fallback: if email is unavailable, show a generic person icon (User from lucide, 16px, white)
```

**Implementation:** Use shadcn `<Avatar>` with `<AvatarFallback className="bg-neutral-950 text-white text-base font-semibold">`. No `<AvatarImage>` — there is no profile image in Phase 2. Phase 4+ may add Gravatar or upload; fallback contract stands.

### Q4 — Empty/placeholder card on dashboard: How to communicate "coming soon" without being cheap?

**Decision:** A single editorial card with dashed border, no shadow, containing a concise typographic message. No emoji, no illustration, no "Construction" iconography. The card speaks through restraint.

Rationale: "Coming soon" cards with emojis or clip-art icons undermine the premium editorial positioning established in Phase 1. A card that communicates future promise through confident copy and generous whitespace feels intentional rather than incomplete. This is also aligned with D-04 (spacious + editorial).

**Stub card contract:**
```
Container: max-w-[480px] (centered in page body on desktop; full-width on mobile)
Border: 1px dashed neutral-200 (dashed distinguishes it from a real content card)
Radius: 12px (--radius, matches pricing cards)
Padding: 32px (p-8)
Background: neutral-50 (bg-neutral-50)
Shadow: none (stub state — not yet a real card)

Content:
  - Overline (label, muted, tracking-widest, uppercase 14px / 0.875rem): "SEGERA HADIR"
  - Heading (heading size 28px, weight 600, mt-3): "Komponen pertama sedang disiapkan."
  - Body (body size 16px, weight 400, muted, mt-4): "Kamu sudah masuk. Komponen segera hadir — pantau terus."
```

**Why tracking-widest on overline:** The all-caps overline label at 14px / 0.875rem (Label role) uses `tracking-widest` (`letter-spacing: 0.1em`) as an exception to the no-tracking rule — small-caps labels conventionally benefit from wide tracking for readability. This is the only instance of wide tracking in Phase 2; it does not extend to other elements.

### Q5 — Error banner on /login?error=link-expired: Inline alert or toast?

**Decision: Inline alert below the form's sub-heading, above the email input.** Not a toast.

Rationale: Toasts are for transient notifications triggered by user action. A link-expired error is a page-load condition — the user navigated to `/login?error=link-expired` because their magic link failed. The error is structural to the page state, not an ephemeral event. Placing it inline (below the page sub-heading, above the input) ensures it cannot be missed and cannot auto-dismiss. Toasts can auto-dismiss or be missed on slow connections.

**Link-expired alert contract:**
```
Placement: inside the login card, below the sub-heading ("Kami akan kirim link masuk ke emailmu."), 
           above the email input — separated by mt-4 from sub-heading and mb-4 from the label
Container: rounded-lg (8px), border 1px solid red-200, bg-red-50, px-4 py-3
Icon: AlertCircle from lucide (16px, text-red-600, flex-shrink-0, mr-2 inline with text)
Text: body size (16px), weight 400, text-red-600
Presence: only rendered when URL search param `?error=link-expired` is present (server-read)
          Absent from the DOM entirely when param is not present
Role: role="alert" aria-live="assertive"
```

This alert does not appear for the generic "Cek email kamu" success flow — that uses a separate success state (see Copywriting Contract).

## Interaction States

### LoginForm submit button (extended from Phase 1)

| State | Visual |
|-------|--------|
| Idle (has text) | `neutral-950` fill, white "Kirim Magic Link", weight 600. Identical to Phase 1 resting. |
| Idle (empty form) | Same as above — button is never visually disabled until user attempts submit |
| Hover (idle) | `neutral-800` fill, 150ms ease-out. From P1. |
| Focus-visible | 2px `outline-neutral-950` with 2px offset. From P1. |
| Loading | `neutral-800` fill, `Loader2` (16px, `animate-spin`) + "Mengirim...", `disabled`, `cursor-not-allowed`, email input also `disabled` |
| Success | Form unmounts, replaced by `<SuccessBlock>` — no button in success state |
| Error (Server Action error — e.g. network) | Form remains; error message "Ups, ada masalah teknis. Coba lagi." appears below the button in `text-red-600` at label size. Button re-enables. (See Copywriting Contract.) |

### Avatar + dropdown trigger

| State | Visual |
|-------|--------|
| Resting | Avatar circle (`neutral-950`, white initial) + user email (muted, label size) + `ChevronDown` (neutral-500, 16px). No background. |
| Hover | Background: `neutral-100` rounded-lg (`rounded-lg` = 8px). Entire right-slot group (`email + avatar + chevron`) gets the hover background. Transition: 150ms. |
| Focus-visible | 2px `outline-neutral-950` offset 2px on the trigger button wrapping the right slot. |
| Active (dropdown open) | `ChevronDown` rotates 180° (`rotate-180`), same background as hover. `aria-expanded="true"`. |

### Dropdown menu ("Keluar")

| State | Visual |
|-------|--------|
| Menu resting | White background, 1px `border-neutral-200`, 8px radius, shadow elevation token, `min-w-[160px]`, `py-1` |
| "Keluar" item resting | Body size, weight 400, `text-neutral-950`, `px-4 py-2` |
| "Keluar" item hover | `bg-neutral-50`, no text color change. Transition: instant (editorial — no fade). |
| "Keluar" item focus-visible | 2px `outline-neutral-950` inset (Radix/base-ui handles this via `data-highlighted`) |
| After click | Dropdown closes, Server Action fires, redirect to `/` |

**Dropdown does not include a confirmation dialog for logout.** Logging out is not destructive (user can request a new magic link immediately from `/login`). The "Keluar" item acts immediately on click.

### DashboardNav (no mobile hamburger in Phase 2)

DashboardNav does not render a mobile hamburger in Phase 2. At the minimal stub stage, the only navigation element is logout — accessible via the avatar dropdown on all viewport widths. The email truncates with `truncate max-w-[140px]` on mobile to prevent overflow.

**Phase 5 forward note:** When Phase 5 adds sidebar navigation, a hamburger slot should be added to the left side of DashboardNav between the logo and the spacer.

## Accessibility Contract

Inherited from Phase 1. Phase 2 extensions:

| Requirement | Commitment |
|-------------|------------|
| Avatar color contrast | White initial on `neutral-950` = 19.5:1 (AAA). Same as body text. |
| Dropdown keyboard nav | shadcn `<DropdownMenu>` via base-ui handles arrow keys, Enter, Escape, focus trap. No custom implementation needed. |
| Link-expired alert | `role="alert"` + `aria-live="assertive"` — screen readers announce on page load since it's a page-load condition. |
| Loading spinner | `Loader2` element has `aria-label="Memuat"` and `role="status"`. Button has `aria-disabled="true"` during load. |
| Success state | Form is replaced; focus moves to the success heading via `useEffect` that calls `.focus()` on the heading (which has `tabIndex={-1}`). Screen readers read the new content. |
| DashboardNav touch target | Avatar+email+chevron trigger wraps in a `<button>` of minimum 44px height. |
| Skip link | Inherited from Phase 1: `<a href="#main">Lewati ke konten utama</a>` first focusable element. Dashboard layout also gets a `<main id="main">`. |
| Language attribute | `<html lang="id">` — already corrected in Phase 1 execution. Confirm present. |
