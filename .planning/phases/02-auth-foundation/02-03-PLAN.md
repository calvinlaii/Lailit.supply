**STEP 1: Create `src/app/(dashboard)/layout.tsx`** — Route group layout with auth gate:

```typescript
import { redirect } from 'next/navigation'
import { getUser } from '@/lib/dal'
import { DashboardNav } from '@/components/dashboard/dashboard-nav'

export default async function DashboardLayout({
  children,
}: Readonly<{ children: React.ReactNode }>) {
  const user = await getUser()
  if (!user) redirect('/login')

  return (
    <>
      <a
        href="#main"
        className="sr-only focus:not-sr-only focus:fixed focus:top-4 focus:left-4 focus:z-[100] focus:bg-white focus:px-4 focus:py-2 focus:text-sm focus:font-semibold focus:text-neutral-950 focus:border focus:border-neutral-200 focus:rounded-lg"
      >
        Lewati ke konten utama
      </a>
      <DashboardNav email={user.email ?? ''} />
      <main id="main" className="flex-1">
        {children}
      </main>
    </>
  )
}
```

This is a Server Component (no 'use client'). `getUser()` from DAL calls supabase.auth.getUser() — JWT-verified. `redirect('/login')` throws a Next.js redirect (MUST NOT be in a try/catch). No `<Footer />` in the dashboard layout.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Unauthenticated request → /dashboard/* | proxy.ts + layout.tsx double-gate: proxy calls updateSession (refreshes token), layout checks getUser() |
| Client Component (UserMenu) → Server Action (signOut) | Form POST via Server Action — browser submits form, no client-side auth state involved |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Elevation of Privilege | src/app/(dashboard)/layout.tsx | mitigate | getUser() from DAL verifies JWT server-side before rendering any dashboard content. redirect('/login') on null — unauthenticated users never see dashboard HTML. |
| T-02-11 | Spoofing | src/app/(dashboard)/dashboard/actions.ts signOut | accept | signOut calls supabase.auth.signOut() which invalidates the server-side session. Replaying the signOut action simply calls signOut on an already-expired session — harmless. |
| T-02-12 | Information Disclosure | src/app/(dashboard)/dashboard/page.tsx | accept | Dashboard page only displays user.email (already known to the user). No sensitive data (membership tier, payment info) exposed in Phase 2. |
| T-02-13 | Elevation of Privilege | Double auth check (proxy + layout) | mitigate | proxy.ts runs updateSession on /dashboard/* (token refresh). layout.tsx calls getUser() (JWT verification). Defense in depth: both layers must pass. |
</threat_model>
